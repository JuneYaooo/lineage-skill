# Skill Roles

`lineage-skill` separates course distillation from Skill generation.

The pipeline first builds a generic `CoursePackage`. A generated Skill then chooses a **role**. Course scope, evidence strictness, and learning progress are separate options; they are not parallel roles.

## Main Roles

| Role | User Intent | Typical Outputs |
| --- | --- | --- |
| `mentor` | "带着我学完这套课。" Guided learning, practice, review, progress-aware study, and application. | learning plan, review sessions, follow-up questions, weak-point diagnosis |
| `expert` | "这门课怎么讲？出处在哪？" Accurate course explanation and source lookup. | concept explanation, lesson lookup, source-backed answers |
| `consultant` | "用这套课的方法分析我的情况。" Private consulting or scenario diagnosis based on course methods. | diagnosis, assumptions, options, recommendations |
| `practitioner` | "直接帮我干活。" Turn course methods into work artifacts. | SOPs, checklists, playbooks, templates, drafts |
| `custom` | "我要一个特殊用途的 Skill。" User-defined course-backed workflow. | custom role files and workflows |

Default role: `mentor`.

## Other Dimensions

These dimensions can combine with any role.

| Dimension | Values | Meaning |
| --- | --- | --- |
| Scope | `single-course`, `multi-course`, `fused` | Whether the package contains one course, several distinguishable courses, or a synthesized course system. |
| Evidence | `standard`, `strict` | Whether answers use normal source grounding or strict audit-style citation. |
| Progress | `none`, `tracked` | Whether the Skill maintains learning progress, weak points, review queues, and plan adjustments. |

Scope should live in package/manifest metadata, not in the Skill name. A Skill can start from one course and later include more courses without renaming.

The builder writes `scope`, `evidence_strategy`, and `progress_strategy` into `lineage_manifest.json`. Use these fields to distinguish single-course, multi-course, fused, strict-citation, and progress-aware variants without changing the public Skill name.

## Naming

If `--skill-name` is omitted, the builder uses:

```text
<course-or-domain-slug>-<role>-lineage
```

Examples:

```text
shanghan-mentor-lineage
ip-camp-consultant-lineage
business-courses-practitioner-lineage
tcm-expert-lineage
```

The final `-lineage` suffix marks Skills generated by `lineage-skill`.

## Commands

Default mentor:

```bash
python scripts/build_course_skill.py \
  --source-dir ./example-course \
  --course-name example-course \
  --mode mentor \
  --output-dir ./dist
```

Private consultant:

```bash
python scripts/build_course_skill.py \
  --source-dir ./example-course \
  --course-name example-course \
  --mode consultant \
  --output-dir ./dist
```

Practical work assistant:

```bash
python scripts/build_course_skill.py \
  --source-dir ./example-course \
  --course-name example-course \
  --mode practitioner \
  --output-dir ./dist
```

Custom role:

```bash
python scripts/build_course_skill.py \
  --source-dir ./example-course \
  --course-name example-course \
  --mode custom \
  --description "A course-backed Skill for sales role-play and weekly pipeline review." \
  --output-dir ./dist
```

## Multi-Course Packages

For multiple courses, first create a combined package:

```bash
python scripts/build_multi_course_package.py \
  --course course-a \
  --course course-b \
  --combined-name business-courses \
  --output-dir .lineage/courses/business-courses
```

Then generate any role from that combined package:

```bash
python scripts/build_course_skill.py \
  --source-dir .lineage/courses/business-courses \
  --course-name business-courses \
  --mode consultant \
  --output-dir ./dist
```

The role is independent of whether the package is single-course or multi-course.
